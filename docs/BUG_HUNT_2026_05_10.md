## #1 — `np.sqrt(int*int*int*int)` (sibling of fix `90bbecb`)

**Pattern**:
```python
tp = int(...)  # ~5K
tn = int(...)  # ~6M for all-Canada
fp = int(...)
fn = int(...)
denom = np.sqrt((tp+fp) * (tp+fn) * (tn+fp) * (tn+fn))   # ✗
```

**Failure mode**: tn ≈ 6M → 4-product ≈ 1e26, exceeds int64 (~9e18) → silent
overflow. Even when in range, `np.sqrt` cannot operate on Python `int`:
```
TypeError: loop of ufunc does not support argument 0 of type int
           which has no callable sqrt method
```

**Affected files audited** (all `*compute*metric*.py` + similar):
- `scripts/compute_full_metric_card.py:371` — already fixed in `90bbecb`
- `scripts/compute_unified_metrics.py:154-158` — already correctly uses `float(...)`
- **`scripts/compute_full_metrics.py:129-133`** — same bug, dead code (no callers found via grep), **fixed defensively** in this session

**Why "dead code" still matters**: someone (or me, next session) might revive
`compute_full_metrics.py` thinking it's the canonical script. Identical
function names mean the bug is easy to copy-paste back into the live one.

## #2 — fire_clim future-year leak (sibling of train_v3.py:2002 fix)

**Pattern**:
```python
nearest = min(fire_clim_arrays.keys(), key=lambda y: abs(y - yr))   # ✗
```

For target year `yr` not directly in `fire_clim_arrays`, this picks the
year with smallest distance — could be `yr+1` (FUTURE). Using a fire-climatology
that includes the target year's fires = label leakage during training.

**Affected file**: `src/data_ops/processing/consolidate_encoder_data.py:279`

This script builds the 1.1TB encoder cache used by 16ch experiments. **Whether
the bug actually triggered depends on whether years 2000-2025 were all
present in `fire_clim_arrays`** (in which case the `yr in dict` branch always
hits and the buggy fallback never runs).

**Audit**: `fire_clim_arrays` is populated by `glob("fire_clim_upto_*.tif")`
which loads all 26 years in `data/fire_clim_annual_nbac/`. So in practice
the fallback branch is unreachable for `yr in [2000, 2025]`. **No data
contamination has occurred** with current data.

**Risk**: if anyone runs this script with a partial fire_clim_dir
(e.g. only training years), the bug becomes live. **Fixed defensively** to
mirror train_v3.py's `max([y for y in keys if y <= yr])` pattern.

## #3 — SLURM `echo exit=$?` masks Python crashes (root cause of metric_card silent fail)

**Pattern**:
```bash
$PYTHON -u -m something
echo "=== Done $(date) exit=$? ==="
```

Bash exit code = last command = `echo` = always 0. SLURM sees the job
"COMPLETED" with elapsed = real walltime even if Python crashed in the
first second. **This is exactly what hid the metric_card sqrt-bug for
hours**: 11 jobs reported COMPLETED in 27 seconds, all log-files 870 bytes.

**Fix**: capture exit code first, re-raise it:
```bash
$PYTHON ...
PY_EXIT=$?
echo "=== Done $(date) exit=$PY_EXIT ==="
exit $PY_EXIT
```

**Affected files (6, all fixed this session)**:
- `slurm/ensemble_logit_narval.sh`
- `slurm/full_metric_card_narval.sh`
- `slurm/prep_13ch_nbac_nfdb_narval.sh`
- `slurm/train_v3_12y_dmodel_sweep_narval.sh`
- `slurm/train_v3_12y_sub4x4_climsim_narval.sh`
- `slurm/train_v3_22y_dmodel_sweep_narval.sh`

**Caveat**: jobs already submitted with the old script (60734864 ensemble_logit,
60734980-990 metric_cards) still have the buggy script — the fix only
helps re-submits. Manually verify outputs of those jobs before trusting them.

## #4 — `reproject(src_nodata=np.nan)` edge holes

**Pattern**: documented in MEMORY.md. `src_nodata=np.nan` blocks bilinear
extrapolation past WGS84 bbox → up to 24% of EPSG:3978 corners get nan.

**Files still using this pattern (NOT fixed in this session):**
| File | Channel | In SOTA? | Risk |
|---|---|---|---|
| `process_cdem_slope.py` | slope | YES (9ch SOTA) | low — converts NaN to 0; ocean edges |
| `process_population.py` | population | YES (9ch SOTA) | low — `np.where(isfinite, x, 0.0)` |
| `process_modis_ndvi.py` | NDVI | YES (13ch/16ch) | medium — used in caches |
| `ecmwf_to_fwi.py` / `_batch.py` | FWI | YES (all training) | medium — could affect FWI edges |
| `process_s2s_to_tif.py` | S2S decoder data | YES (s2s_legacy decoder) | medium |
| `process_srtm_slope.py` | alt slope | unused (SRTM is dead code) | n/a |

**Why not fixed now**:
- Re-running these scripts means rebuilding 309GB+ caches → days
- Affected static channels (population/slope) clip NaN to 0 downstream → only edges, not interior, are affected
- Audit script `scripts/audit_data_complete.py` should catch any catastrophic edge corruption
- Existing 8.07× SOTA was trained on this data — implies impact is bounded

**Action**: leave as tech debt. If we ever rebuild caches (e.g. for a paper
reproducibility check), apply the fix from `slurm/process_era5_narval.sh`'s
fix and re-run the audit.

## #5 — sub4x4 decoder collate (KNOWN, NOT FIXED)

```
RuntimeError: stack expects each tensor to be equal size,
  but got [32, 128] at entry 0 and [32, 136] at entry 6
```

The sub4x4 dataset returns variable-shape tensors per sample (128 vs 136-dim).
Likely the climsim sample-weight column is being concat'd into the feature
vector for some samples but not others.

**Why not fixed in this session**: needs a careful read of the sub4x4 dataset
+ collate functions, not a 1-line patch. Worth doing in a fresh window with
focus.

**Why it matters**: this is the only architectural axis we haven't truly
tested. s2s_legacy decoder context is 9–12 dim; sub4x4 is 128 dim — **14×
more information density**. Could unlock another double-digit % gain.

## Lessons added

1. **`echo "$?"` is poison in SLURM scripts**. Always capture into a variable
   first, then `exit` it. Audited entire `slurm/` and fixed all 6 occurrences.

2. **Dead code with the same function name as live code is a rake**. A grep
   for the next bug finds two files; if you fix the wrong one, the live
   bug stays. Strategy: fix both, or delete dead code immediately on
   discovery — don't leave it.

3. **Defensive year-filter pattern**: any time you see `min(years, key=abs(y-yr))`,
   replace with `max([y for y in years if y <= yr])`. The "nearest year" idiom
   is almost always wrong in time-series leakage contexts.
